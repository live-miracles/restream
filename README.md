# Restream

Restream is a host-run streaming control plane built on [MediaMTX](https://github.com/bluenviron/mediamtx). It manages stream keys, pipelines, output destinations, and FFmpeg jobs from a browser dashboard.

## How It Works

1. A publisher sends RTMP or SRT ingest to MediaMTX.
2. Restream stores stream keys, pipelines, outputs, job state, and logs in local SQLite (`data.db`).
3. When an output starts, Restream probes the MediaMTX path, spawns FFmpeg pulling from MediaMTX, and tracks the process.
4. The dashboard reads `/config` and `/health` to show pipeline state, output state, system metrics, and logs.

MediaMTX owns media routing. Restream owns orchestration and state.

## Local Host Run

Install Go (1.23+) if not already present:

```sh
# Download from https://go.dev/dl/ and follow the install instructions
go version  # verify
```

Install Node.js dependencies (dev toolchain — TypeScript, Tailwind, hls.js):

```sh
npm ci
```

Build frontend assets, then compile the Go binary (frontend is embedded in the binary):

```sh
make build
# which runs: npm run ts-build && npm run css && npm run vendor-hls && go build
```

Or step by step:

```sh
npm run ts-build      # compile public/ts/ → public/js/
npm run css           # compile public/input.css → public/output.css
npm run vendor-hls    # copy hls.min.js → public/vendor/
go build -o dist/restream ./cmd/server   # embed everything into the binary
```

Start MediaMTX with the checked-in config:

```sh
./mediamtx
```

In another terminal, start the backend:

```sh
./dist/restream
```

For development mode (run each in its own terminal):

```sh
npm run ts-watch    # recompile TypeScript on change
npm run css-watch   # recompile CSS on change
# After any frontend change, rebuild and restart:
make build && ./dist/restream
```

Useful frontend commands:

```sh
make frontend        # compile TS + CSS + copy hls.js (must run before go build)
npm run format       # run prettier
npm run format:check # check formatting without writing
npm run test:normalization  # unit tests for URL normalization helpers
npm run test:integration    # 2x3 end-to-end test (requires app + MediaMTX running)
```

The dashboard runs on `http://localhost:3030/` by default. Set `PORT=<n>` to use a different port.

## Runtime Files

- `data.db`: local SQLite database (set `DB_PATH` env var to change location)
- `dist/restream`: compiled Go binary (self-contained — all frontend assets are embedded)

The following are build-time intermediates, gitignored, and recreated by `make frontend`:
- `public/js/`: compiled from `public/ts/`
- `public/vendor/`: vendored hls.js (copied from node_modules)
- `public/output.css`: compiled from `public/input.css`

## Code Map

- `cmd/server/main.go`: application entry point — wires services, routes, and static files
- `internal/api/`: HTTP route handlers (config, pipelines, outputs, encodings, metrics, preview proxy)
- `internal/services/`: health collection (`health.go`), output/FFmpeg lifecycle (`outputs.go`), startup timers (`bootstrap.go`)
- `internal/db/`: SQLite schema and query helpers
- `internal/etag/`: ETag management for `/config` 304 responses
- `internal/ffmpeg/`: FFmpeg argument builders, URL validation, encoding presets
- `internal/mediamtx/`: MediaMTX API client (stream keys, ingest URLs, path builders)
- `internal/progress/`: shared in-memory FFmpeg progress state (total_size, bitrate)
- `internal/apputils/`: validation, token masking, structured logging
- `public/ts/`: TypeScript source for the dashboard frontend
  - `public/ts/types.ts`: shared type definitions for all API shapes and view models
  - `public/ts/global.d.ts`: `Window` augmentation and HLS.js interface declarations
  - `public/ts/core/`: state, API client, pipeline data transformation, and utilities
  - `public/ts/features/`: dashboard rendering, editor modals, input preview, and metric display
  - `public/ts/history/`: output and pipeline log/history subsystem
- `public/js/`: compiled output from `public/ts/` (generated by `npm run ts-build`)
- `public/`: dashboard HTML, compiled CSS
- `test/`: behavior and normalization tests
## Testing

### Unit Tests

```sh
npm run test:normalization
```

Tests URL helper functions inside `test/artifacts/run-2x3.mjs` (the integration test harness, not the backend). Requires Node but no running services.

### Integration Test (2x3)

The 2x3 test starts RTMP and SRT input publishers for two pipelines, starts all six outputs, waits for everything to reach `on` status, then stops the outputs. It requires a running app stack and MediaMTX.

Build and start the stack first:

```sh
make build
./mediamtx &
./dist/restream &
```

Then run the test:

```sh
make run-2x3
# or: npm run test:integration
```

The test reads from `test/colorbar-timer.mp4`, uses the manifest at `test/artifacts/session-2x3-manifest.json`, and writes logs to `test/artifacts/logs/`.

Environment overrides:

| Variable | Default | Description |
|---|---|---|
| `API_URL` | `http://localhost:3030` | Backend base URL |
| `MANIFEST_PATH` | `test/artifacts/session-2x3-manifest.json` | Path to manifest |
| `INPUT_PROTOCOLS` | `rtmp,srt` | Comma-separated ingest protocols |
| `TIMEOUT_SEC` | `120` | Seconds to wait for all streams to go green |
| `KEEP_RUNNING` | `0` | Set to `1` to leave publishers and test resources in place |

### CI

GitHub Actions (`.github/workflows/ci.yml`) runs jobs on every push and pull request:

- **format** — Prettier check (`npm run format:check`)
- **unit** — normalization test (`npm run test:normalization`)
- **build** — `make frontend && go build` to verify the binary compiles cleanly
- **integration** — 2x3 end-to-end test; only runs after unit+build pass; builds the binary, starts MediaMTX and the app, runs the test; uploads `test/artifacts/logs/` as an artifact on failure

## Linux VM Deployment (GCP)

The binary is self-contained (frontend assets embedded). The server only needs FFmpeg and MediaMTX — no Go, no Node, no npm.

### Server Requirements

- 2 vCPU, 4 GB RAM, 20 GB disk
- Ubuntu 22.04 or 24.04 with systemd
- FFmpeg 7.1+ and MediaMTX — installed by the setup script

Access is via IAP tunnel — direct SSH is disabled. All commands use `gcloud compute ssh/scp --tunnel-through-iap`.

### Step 1 — Set up the server (once)

Clone the repo onto the VM (for `mediamtx.yml` and the setup script), then run:

```sh
gcloud compute ssh my-vm --tunnel-through-iap -- \
  "sudo git clone https://github.com/live-miracles/restream /opt/restream && \
   sudo bash /opt/restream/scripts/server-setup.sh"
```

This installs FFmpeg, MediaMTX, creates the `restream` service user, and registers systemd units. It does **not** build anything — no Go or Node needed on the server.

Add `--zone=ZONE --project=PROJECT` to any `gcloud` command if needed.

### Step 2 — Build and deploy from your dev machine

```sh
bash scripts/deploy.sh my-vm
# with zone/project:
bash scripts/deploy.sh my-vm --zone=us-central1-a --project=my-project
```

This cross-compiles a Linux amd64 binary (frontend embedded), copies it to the VM via IAP, and restarts the service.

### GCP Firewall Rules

Open these ports in your VPC firewall (VPC Network → Firewall):

| Port | Protocol | Purpose |
|---|---|---|
| `3030` | TCP | Dashboard (or put a reverse proxy on 80/443 instead) |
| `1935` | TCP | RTMP ingest |
| `8890` | UDP/TCP | SRT ingest |

MediaMTX API (`9997`) and HLS preview (`8888`) stay localhost-only.

### Settings

To change the server name or manage custom encodings, open the Settings page: `http://<VM-external-IP>:3030/settings.html`.

To update the MediaMTX config, edit `mediamtx.yml` locally and copy it to the server:

```sh
gcloud compute scp mediamtx.yml my-vm:/opt/restream/ --tunnel-through-iap
gcloud compute ssh my-vm --tunnel-through-iap -- \
  "sudo cp /opt/restream/mediamtx.yml /etc/restream/ && sudo systemctl restart mediamtx"
```

### Updating

```sh
bash scripts/server-update.sh my-vm
```

Rebuilds frontend + binary, copies to VM, restarts `restream.service`.

### Operations

```sh
# Logs
gcloud compute ssh my-vm --tunnel-through-iap -- journalctl -u restream.service -f
gcloud compute ssh my-vm --tunnel-through-iap -- journalctl -u mediamtx.service -f

# Restart
gcloud compute ssh my-vm --tunnel-through-iap -- sudo systemctl restart restream.service

# Health check
gcloud compute ssh my-vm --tunnel-through-iap -- curl -fsS http://127.0.0.1:3030/healthz

# Backup database
gcloud compute scp --tunnel-through-iap \
  my-vm:/var/lib/restream/data.db ./data.db.bak-$(date +%F)
```

### Reverse Proxy and TLS

Put a reverse proxy (nginx, Caddy) in front of port `3030` and terminate TLS at `443`. Keep MediaMTX API (`9997`) bound to localhost. Restrict direct access to port `3030` via firewall once the proxy is in place.

### Security Baseline

- Both services run as non-root (`restream` user).
- Keep `9997` and `8888` local-only.
- Use firewall rules to restrict ingest and UI ports.
- Rotate stream keys periodically.
- Apply OS and package updates on a maintenance schedule.
